# Loyalty Strategy Input (Owner Material)

Date logged: Friday, 2026-03-06
Status: Input for planning, not a locked decision

## Why this note exists

This note captures new owner-side commercial thinking that should guide loyalty requirements for ROLLER and Patch.

## Core business direction

1. Keep base pricing strong; use loyalty to increase repeat visits.
2. Move behavior from occasional visits (1-3/year) toward frequent visits (4-12/year).
3. Use pricing to manage capacity and loyalty to drive return behavior.

## Pricing and offer logic to reflect in loyalty planning

1. Anchor ticket is 2h (1h only in small parks).
2. Upsell path should support +1h and full day.
3. Peak periods can use adult=child pricing and premium logic.
4. Off-peak should reduce friction with parent/guardian bundle logic.
5. Family bundles are preferred over blanket discounts.
6. Socks policy is a monetization and sustainability lever.

## Customer logic to support

1. Main revenue child segment: 5-14 years.
2. Parent/guardian roles must be supported in pricing and benefits.
3. First-visit conversion can include immediate loyalty benefit.
4. Loyalty should enable exceptions without weakening base price.

## Platform operating model (global vs local)

Global rules (shared):
1. Anchor ticket logic.
2. Core child/parent logic.
3. Socks policy principles.
4. Upsell path logic.

Local adaptation (park/country):
1. Price level.
2. Seasonal campaigns.
3. Bundle details.
4. Local member surprises.

## Loyalty concept variants from material

1. Variant A: paid membership model with park-specific perks.
2. Variant B: free join + instant welcome reward + milestone rewards.
3. Variant C: account-first + milestone rewards + local surprises.

Common requirement across variants:
1. Visit/progress tracking.
2. Milestone reward logic (example target: reward after 6 qualifying visits).
3. Local benefit layer.

## What this means for ROLLER/Patch requirements

1. Support account identity by email and phone.
2. Support qualifying-visit counting and milestone rewards.
3. Support immediate welcome reward rules.
4. Support parent/guardian benefit exceptions.
5. Support global rule templates plus park-level benefit overrides.
6. Support POS-friendly redemption and clear audit trail.
7. Support first-visit capture and repeat-visit measurement.

## Visitor value statement to test

1. "Why should I come back?" -> clear progress + meaningful next reward.
2. "Why should I stay active?" -> ongoing member benefits and local surprises tied to repeat visits.
